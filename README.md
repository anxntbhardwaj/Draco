## Introduction

Draco is a command‑line tool written in Python that performs high‑volume HTTP GET flooding against a specified target web server. It is designed for **educational and authorised testing** – primarily to assist system administrators, security researchers, and students in understanding the dynamics of application‑layer denial‑of‑service (DoS) attacks. The utility can be executed on Android devices via the Pydroid 3 environment as well as on conventional desktop operating systems (Windows, Linux, macOS).

The program creates multiple concurrent threads, each of which repeatedly sends HTTP requests to a user‑supplied URL. To simulate legitimate browser traffic, each request is issued with a randomised `User‑Agent` header. This approach helps bypass elementary request‑filtering mechanisms that rely on static identifiers.

Although Draco can generate substantial traffic volumes, its effectiveness is constrained by the capabilities of the host machine and the presence of protective measures on the target side. The tool does not employ IP spoofing, proxy rotation, or distributed infrastructure – it operates from a single source address. Therefore, its usage is inherently traceable and must be restricted to environments where explicit authorisation has been granted.

## What is an Application‑Layer DoS Attack?

A denial‑of‑service attack aims to make a service unavailable to legitimate users by exhausting the target's resources. These resources may include network bandwidth, CPU cycles, memory, database connections, or file descriptors. Attacks are commonly categorised by the OSI layer they target:

| Category | Layer | Common Techniques | Impact |
|----------|-------|-------------------|--------|
| Volume‑based | Network (L3) | UDP floods, ICMP floods | Saturates bandwidth (Gbps) |
| Protocol‑based | Transport (L4) | SYN floods, ACK floods | Exhausts connection state tables |
| Application‑layer | Application (L7) | HTTP floods, slow‑loris | Consumes server CPU and database resources |

Draco focuses on the **application layer (L7)** , specifically the HTTP GET flood. This type of attack is particularly insidious because the requests appear similar to those generated by actual users, making them difficult to distinguish from normal traffic. A relatively modest request rate can overwhelm a server that processes dynamic content (e.g., PHP, Python, or database‑driven applications) by forcing it to execute computationally expensive operations for each request.

The primary goal of an HTTP flood is not to saturate the network pipe but to **exhaust the server's ability to respond**. This is achieved by:

- Consuming all available worker processes or threads in the web server (Apache's `MaxRequestWorkers`, Nginx's worker processes).
- Overloading backend interpreters (PHP‑FPM, uWSGI) with requests that require significant processing.
- Filling database connection pools, leading to "too many connections" errors.
- Triggering high CPU utilisation due to TLS handshake overhead and header parsing.

Understanding these mechanisms is essential for building effective defences.

## Architectural Overview

Draco is implemented as a single Python script with minimal external dependencies. The code is organised into the following functional segments:

1. **Initialisation**  
   - Import of standard and third‑party modules: `threading`, `requests`, `os`, `time`, `sys`, `webbrowser`, `pyfiglet`, `fake_useragent`.
   - Definition of ANSI colour codes for terminal output.
   - Display of an ASCII banner via `pyfiglet`.

2. **Dependency Management**  
   - The script assumes that required packages (`requests`, `fake_useragent`, `pyfiglet`) are installed. Installation instructions are provided in the documentation; on Pydroid 3, the integrated package manager can be used to install these libraries with a single command.

3. **Request Sender Function (`send_request`)**  
   - Accepts a target URL.
   - Generates a random User‑Agent string using `UserAgent().random` for each request.
   - Issues a `GET` request with `requests.get()` inside a `try`‑`except` block.
   - Prints success or error messages to the console with colour coding.

4. **Thread Launcher (`launch_attack`)**  
   - Takes the target URL and the number of threads as parameters.
   - Creates and starts daemon threads, each running `send_request`.
   - Daemon threads terminate automatically when the main process exits.

5. **Main Entry Point**  
   - Collects user input: target URL and thread count.
   - Opens the author's Instagram page (this feature can be removed by commenting out the corresponding line).
   - Initiates the attack.

The code intentionally remains straightforward to facilitate comprehension and modification for educational purposes. Advanced features, such as custom headers, proxy support, or throttling, are omitted to keep the core logic clear.

## Core Mechanisms

### 4.1 Multi‑threading and Concurrency

Draco uses Python's `threading` module to achieve concurrency. Each thread runs an infinite loop that issues HTTP requests. Because network I/O releases the Global Interpreter Lock (GIL), multiple threads can execute nearly in parallel, achieving near‑linear scaling up to the limits imposed by the operating system and hardware.

Threads are created as **daemons** (`thread.daemon = True`) to ensure they are terminated when the main program exits, preventing orphaned processes and resource leaks.

The number of threads is user‑configurable. However, excessive thread creation can lead to resource exhaustion on the attacker's side – each thread consumes memory (stack space) and incurs scheduling overhead. The optimal thread count depends on the host's CPU core count, available RAM, and network bandwidth.

### 4.2 User‑Agent Rotation

The `fake_useragent` library maintains a curated list of real‑world User‑Agent strings for major browsers and operating systems. By calling `UserAgent().random`, Draco obtains a new, plausible string for each request. This technique reduces the likelihood of being blocked by simple filters that inspect the `User‑Agent` header.

However, this rotation does not guarantee evasion. More sophisticated detection systems monitor additional attributes such as request frequency, connection behaviour, and TLS fingerprint (for HTTPS), which are beyond the scope of this tool.

### 4.3 Request Lifecycle

For each iteration, the thread performs the following sequence:

1. Resolve the target domain (handled by the `requests` library).
2. Establish a TCP connection to the server (or reuse a session, though Draco opens a new connection each time).
3. For HTTPS, perform a TLS handshake.
4. Send the HTTP GET request line and headers.
5. Wait for the server's response (headers and body).
6. Close the connection (the `requests` library does this automatically when the response object is discarded).

The overhead of connection establishment (TCP handshake and TLS handshake) contributes significantly to the server's resource consumption, especially when repeated thousands of times per second.

### 4.4 Error Handling

Exceptions that occur during request processing are caught and logged with an error indicator. The thread then continues to the next iteration without interruption. This makes the attack resilient to transient network issues and server downtime.

Commonly encountered errors include:
- `requests.exceptions.ConnectionError` – DNS resolution failure, refused connection, or network unreachable.
- `requests.exceptions.Timeout` – The server did not respond within the default timeout.
- `requests.exceptions.TooManyRedirects` – Redirect loop or excessive redirections.

All exceptions are suppressed to ensure the attack continues unabated.
